**Goal:** Localize the complete reachable Settings workspace, including the
current Investor Profile panel and Settings-owned backend-copy mappings, into
`zh-Hant` and English without changing Settings IA, losing drafts or work,
exposing raw diagnostics in normal mode, changing formatter semantics, or
rendering the future language selector.

**Architecture:** Extend the existing bundled `settings` namespace and add two
Settings-only static mapping seams: `settingsCopy.ts` owns closed frontend
domain IDs and registry copy, while `settingsBackendCopy.ts` owns provider,
schedule, storage, API-error, and diagnostic projections. Add typed HTTP
metadata to the existing API client without changing legacy `Error.message`
output, then pass the existing Shell `developerMode` bit into Settings so raw
backend/source diagnostics render only in Developer Mode. Shared Research
presenters keep their current output; Settings consumes new semantic IDs rather
than capturing those source-language labels.

**Tech Stack:** React 18, TypeScript 5.9, i18next 26.3.6 selector API,
react-i18next 17.0.10, Vitest 4/jsdom, the existing TypeScript-AST literal
scanner, Vite/Electron, and Playwright/CDP verification against isolated
temporary profile databases.

## Locked Decisions

1. I18N-2 is frontend-only. Backend routes, DTOs, schemas, stores, schedulers,
   collectors, extension code, desktop-main code, and backend tests remain
   byte-identical to `707a5705` without exception.
2. The public locale selector remains absent. `en` is exercised only through
   test/dev locale setup and isolated profiles. No autonym, disabled selector,
   fourth Settings tab, fake registry anchor, or Shell language control appears.
3. Settings keeps exactly three workflow tabs, nine anchors, one mounted group,
   current tab order, current directory behavior, current search behavior,
   current dirty/busy navigation guard, current Drawer behavior, and current
   active-group persistence.
4. Locale changes rerender copy in place. They do not remount the active group,
   clear search, close a Drawer or disclosure, reset form values, restart OAuth,
   clear discovery/test state, lose pending confirmations, or move focus.
5. The current Investor Profile panel is fully in scope even though P2.8 Slice 5
   later redesigns it. I18N-2 leaves no reachable monolingual Settings panel.
6. `settings` is the only namespace extended. It gains exactly `+485` non-empty
   leaf keys in each locale, for `486` total leaves including the existing
   `locale.writeFailed`. Recursive key paths remain exactly equal.
7. Static selector calls remain the only translation-key style. No key is built
   from a provider, task, source, field, status, error code, user value, template
   string, concatenation, or backend label.
8. `settingsCopy.ts` owns static frontend domain copy: workflow groups, sections,
   bilingual search aliases, model tasks, effort/thinking values, route sources,
   Investor Profile presets/horizons/stances/mismatch states, and related labels.
9. `settingsBackendCopy.ts` owns Settings projections for stable backend IDs and
   codes: provider/config fields, client-ID domains, provider health, provider
   tests, schedule sources/results, body backlog, coverage states, SA Extension
   segments, setup codes, and typed HTTP errors.
10. Known IDs map through exhaustive explicit switches/records. Unknown IDs are
    displayed as stable identifiers beside localized generic copy. Unknown raw
    backend labels, descriptions, reasons, warnings, and details never become
    translation keys and never replace the stable ID.
11. Settings registry structure stores only IDs. Its visible title/description
    come from `settingsCopy.ts`; search values include both locales' static
    titles, descriptions, and aliases, so every pre-I18N-2 title, description,
    and keyword query remains searchable independent of locale.
    `searchSettings("")` remains deterministic registry order.
12. Search metadata is not a new allowlist class. Every section has one static
    `searchAliases` leaf per locale. The pure search helper explicitly reads both
    bundled resource trees and never calls a dynamic translation key.
13. `TaskInfo.label`, `TaskInfo.description`, `EffortOption.label`,
    `EffortOption.description`, provider `label`, schedule `label`/`description`,
    config-field `label`/`guard_reason`, and backend `reason` are not normal-mode
    copy authorities.
14. Model/task copy maps from `ModelTask`, effort ID, thinking-mode ID, source,
    and stable reason code. Model IDs and provider-supplied model names remain
    source values. Unknown effort/thinking/reason values remain stable IDs.
15. `modelPicker.ts` stays shared with Research. It gains semantic group IDs and
    compatibility metadata while retaining its existing `label` behavior for
    Research. Settings renders translated group/suffix copy from the semantic
    fields. I18N-2 does not alter Research DOM or migrate `modelRoutingUx.ts`.
16. `personalizationDisplay.ts` stays shared with AICard/Research and remains
    byte-identical. Investor Profile uses Settings-owned stance/mismatch mapping.
    Calibration messages, proposal rationales, context preview, freeform notes,
    custom edges, avoidances, and behavioral flags remain user/source content.
17. Credential aliases, account labels, masked values, model IDs, ticker symbols,
    FRED series IDs/titles, timestamps, dates, counts, URLs, and provider-returned
    article/data values pass through unchanged.
18. Existing number/date/money/count/duration/timezone formatters remain
    byte-identical. `toLocaleString()`, `formatCount`, `shortDate`,
    `formatSystemTimestamp`, signed/P&L formatting, and units are not moved into
    i18next.
19. Add exported `ApiError extends Error` with `status`, `path`, `code`, and
    `diagnostic` metadata. Existing GET and mutation `.message` strings stay
    byte-for-byte compatible for every unmigrated consumer.
20. Structured FastAPI `detail.code` is the only backend error code. String
    detail and structured `detail.message` are diagnostics, not normal copy.
    Malformed/non-JSON bodies produce `code=null` and `diagnostic=null`.
21. Network and timeout failures remain ordinary `Error` instances. Settings
    maps them to localized generic outcomes; Developer Mode may reveal their
    sanitized `.message`.
22. Normal Settings mode renders no raw `detail`, `warning`, `last_error`,
    exception text, schedule skip reason, OAuth error detail, provider probe
    expected/observed/error text, or assembled generic API `.message`.
23. Developer Mode may render those diagnostics in a clearly labelled
    `Developer diagnostics` disclosure/cell. It does not bypass existing secret
    redaction and does not place diagnostics in `aria-live`.
24. `App.tsx` passes its existing `developerMode` value to `SettingsView`.
    Settings passes it only to owned children that need the diagnostic boundary.
    No second Developer Mode owner or persistence key is introduced.
25. Provider/config test tri-state remains `ok=true`, `ok=false`, `ok=null`.
    Normal mode maps each state without rendering backend detail. Unknown
    provider/config field uses the stable provider/field ID.
26. Schedule skip copy no longer infers `already running` by matching an English
    reason substring. `status="skipped"` maps to a generic localized skipped
    outcome; raw reason is Developer Mode only.
27. Provider disabled/retired/config-missing state remains semantically distinct.
    I18N-2 does not revive PG/migration narration or removed rollout controls.
28. Traditional Chinese remains equivalent except for four reviewed corrections:
    duplicate bilingual headings become one locale-owned label; normal-mode raw
    diagnostics become authored outcomes; backend-owned labels stop overriding
    frontend terms; and skip reasons stop controlling status semantics.
29. The corrected Traditional Chinese headings are `資料來源與排程`, `市場資料`,
    `交易日 / 價格覆蓋`, and `新聞資料`; English uses `Data Sources and Schedules`,
    `Market Data`, `Trading-day / Price Coverage`, and `News Data`.
30. English workflow/section terms are `AI and Models`, `Personalization`,
    `Data and Sync`, `Provider Sign-in and Credentials`, `Model and Task Routing`,
    `Fixed AI Task Runtime Limits`, `AI Research Runtime Limits`,
    `Investor Profile`, `Data Sources and Schedules`, `Market Data`, `News Data`,
    and `Macro Data`.
31. Existing zh-Hant behavior tests continue to assert rendered copy. They are
    not converted to resource-key assertions. English coverage is additive and
    focused by surface family.
32. No CSS, breakpoint, token, IA, control-size, font-size, polling, request,
    mutation, persistence, or data-flow change is pre-authorized. Any genuine
    English overflow triggers the stop-and-amend path before CSS changes.
33. `settings/legacy/AppRecordsSection.tsx` is unreachable and remains
    byte-identical. I18N-2 does not localize, delete, or reactivate it.
34. Design Kit does not sync partial localization. I18N-6 owns selector and
    bilingual release sync.
35. `ARKSCOPE_TERMINOLOGY.md` owns the exact Settings workflow/section,
    Assistant Stance, and risk-mismatch pairs consumed here. The existing
    `maps every Investor stance and mismatch state in both locales` node also
    compares every zh-Hant result byte-for-byte with the byte-identical
    `personalizationDisplay.ts` presenter so two mappings cannot silently
    diverge during the staged migration.
36. `settingsSearchValues()` preserves the complete pre-I18N-2 search surface.
    Each zh-Hant `searchAliases` leaf retains every baseline title,
    description, and keyword value not already present verbatim in the new
    zh-Hant title/description; the helper also reads both locale titles and
    descriptions directly. The existing bilingual-search node loops over every
    baseline title, description, and keyword and requires the same section
    match.
37. Existing `modelPicker.test.ts` assertions are structurally tolerant of
    additive fields, but the ordering and legacy-compatibility nodes are
    strengthened in place to pin the reviewed group `id`, `baseLabel`, and
    `compatibility` metadata. Compatibility entries carry undecorated
    `baseLabel` and explicit provenance into `groupedModelEntries()`; production
    code never reconstructs them by parsing the legacy decorated `label`.
    This is an evolved-node contract, not a new node.

## Independent Plan Review Resolution

Independent written review returned GREEN. Its one required amendment was
technically valid: I18N-2 introduces canonical Settings IA labels and a second
locale-aware owner for Investor enum labels, so a Design Authority that omitted
those pairs would lag the product and permit cross-surface zh-Hant drift.

The review's raw count double-counted D29/D30 overlaps and terms already present
in the authority (`Macro Data`, `Risk appetite`, and `Risk capacity`). The
clearance amendment therefore adds the exact deduplicated set instead:

- 12 Settings workflow/section/subsection pairs not already present;
- all seven Assistant Stance pairs; and
- all four risk-mismatch pairs.

It also strengthens one already-planned table-loop node to compare zh-Hant
stance/mismatch output with `personalizationDisplay.ts`. At that first
clearance, resource, scanner, test-node, backend, and file-count ledgers were
otherwise unchanged; the resource-only correction from deep re-review is
recorded below.

Deep re-review found one arithmetic-bearing baseline error. Production
`HORIZONS` contains exactly five values and its select has no placeholder; the
prior six-choice inventory therefore counted one nonexistent leaf. The exact
resource contract is corrected from `investor 54 / +486 / 487 total` to
`investor 53 / +485 / 486 total`. The other enumerated bases were independently
rechecked and remain unchanged.

The same review raised two advisories. The search advisory is valid: reading
only `searchAliases` would narrow today's title/description/keyword search, so
the contract now includes both locale titles and descriptions plus aliases and
strengthens an existing node against every baseline value. The model-picker
claim that additive fields would necessarily break existing exact assertions
is not technically correct: the current tests project selected fields or use
`objectContaining`. The underlying coverage concern is valid, so those nodes
are explicitly strengthened in place without changing the node ledger.
